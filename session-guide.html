WK 1 URL Planning — Building the Index of 100 Real Websites 3 hours · No code yet ▼
Hr 10:00–1:00
🎙️ Faculty — Lecture + Setup
  • Introduce the Mini Google Search Engine project — show a working demo of the final output (real search results with clickable URLs)
  • Explain the 6-week timeline and all 8 deliverables students will produce
  • Explain what a search engine needs before it can search anything: a defined set of pages (the index)
  • Distribute theme assignment list — each student gets one topic (10 themes, 1 per student): Education & AI, Mohali & Punjab, Technology & Coding, Careers & Business, Lifestyle & Health, Food, Sports, Travel, Science, Arts & Culture
  • Show the LMS projects page — walk through P1 deliverables checklist
✏️ Student — Listen + Note
  • Note assigned theme in personal notebook
  • Open LMS on laptop — navigate to Shadow Projects → P1
  • Write down: what is your theme, what kind of websites will you search for, name 3 example websites you already know
  • Open Notion — create AI diary document using the template shared by faculty
Hr 21:00–2:00
🔍 Faculty — URL Quality Demo
  • Show what makes a URL suitable for indexing: publicly accessible, text-rich, in English, not behind login
  • Live demo: open Wikipedia, GeeksForGeeks, and a blocked site — explain the difference
  • Show how to use ChatGPT / Gemini to suggest URLs by topic — demonstrate prompt: "Give me 10 real Wikipedia URLs about [theme] suitable for a student search engine project"
  • Emphasise: every URL must be manually verified in the browser before adding to the list
  • Show the url-list.txt format: one URL per line, plain text file
🔍 Student — URL Collection
  • Open ChatGPT or Gemini — use the faculty prompt adapted for your theme
  • Manually open every suggested URL in the browser — verify it loads, has text content, is in English
  • Begin building your 10 URLs — paste verified URLs into a text file
  • Log your ChatGPT / Gemini interaction in the Notion AI diary: Tool used, Prompt, Response summary, How you used it
Hr 32:00–3:00
✅ Faculty — Review + Approval
  • Call each student — review their URL list progress
  • Check: are URLs publicly accessible? Are they text-rich? Are they thematically appropriate?
  • Reject any URLs that are paywalled, image-only, or irrelevant
  • Combine all 10 students' lists — show how 100 URLs are assembled into one url-list.txt
  • Share the approved combined url-list.txt with all students via WhatsApp/LMS before end of session
🤖 AI Tool Check
  • Verify students logged their ChatGPT/Gemini interaction in Notion diary
  • Confirm Notion AI diary template is set up correctly for all 6 weeks
📋 Student — Finalise List
  • Finalise your 10 URLs — incorporate faculty feedback
  • Save as url-list.txt in your project folder
  • Verify the Notion AI diary has at least 1 entry logged from today
  • No code written yet — Week 1 is entirely planning and data curation
📄 Week 1 Deliverable: url-list.txt — 100 real URLs, one per line, faculty-approved. Notion AI diary started with minimum 1 entry.
WK 2 Web Crawling — Collecting Data from Real Websites 3 hours · Python · requests · BeautifulSoup ▼
Hr 10:00–1:00
🎙️ Faculty — Crawler Concept
  • Explain what a web crawler is — analogy: Googlebot visits billions of pages; students build a mini version for 100 pages
  • Show the step-by-step flow on projector: url-list.txt → requests.get() → HTML received → BeautifulSoup → extract title/keywords/content → write row to metadata.csv → wait 1.5 seconds → next URL
  • Explain HTTP requests: what happens when your browser visits a webpage vs what requests.get() does
  • Show raw HTML of a Wikipedia page — then show what BeautifulSoup extracts from it (just 3 values from thousands of lines)
  • Explain why HTML files are NOT saved to disk — URL in metadata.csv links directly to live site
✏️ Student — Understand First
  • Open a Wikipedia page in browser — right-click → View Page Source — see raw HTML
  • Identify in the HTML: where is the title tag? Where is the meta keywords tag? Where is the body text?
  • Write in notebook: what 3 pieces of data will the crawler extract from each page, and why?
  • Install libraries if not already done: pip install requests beautifulsoup4
Hr 21:00–2:00
💻 Faculty — Live Crawler Build
  • Open VS Code on projector — build crawler.py live, explaining every line as it is typed
  • Walk through: open url-list.txt → loop over URLs → requests.get() with User-Agent header → BeautifulSoup parsing → extract title, keywords, content → csv.writer → time.sleep(1.5)
  • Explain the User-Agent header — why some sites (like GeeksForGeeks) return 403 without it
  • Run the crawler live on 5 URLs — show metadata.csv being built row by row
  • Show how to handle exceptions: try/except around requests.get() so one failed URL does not stop the whole crawl
💻 Student — Write Your Crawler
  • Open VS Code — create crawler.py in your project folder
  • Type every line yourself — do not copy-paste from projector
  • Run crawler on your own url-list.txt — watch metadata.csv being created
  • If you get a 403 error: ask faculty or use ChatGPT to understand why — then fix it yourself
  • Log any ChatGPT/Gemini interaction used for debugging in your Notion diary
Hr 32:00–3:00
🧑‍🏫 Faculty — Debug Support + CSV Review
  • Circulate — help students fix crawler errors (403 blocks, encoding errors, missing metadata tags)
  • Show on projector: open metadata.csv in Excel — check column quality, look for blank titles or missing content
  • Demonstrate manual popularity scoring: open CSV in Excel, add popularity column, explain score range 30–90 based on how well-known each page is
  • Confirm: every student has a complete metadata.csv with 100 rows before end of session
📊 Student — Review + Score
  • Open your metadata.csv in Excel — check all 100 rows are present
  • Scan for blank titles or missing content — fix by re-running crawler on those URLs
  • Manually assign popularity score (30–90) for each page: 90 = Wikipedia main articles, 60 = well-known sites, 30 = lesser-known pages
  • Save final metadata.csv — this is the complete database for all remaining weeks
📄 Week 2 Deliverable: crawler.py — working web crawler. metadata.csv — 100 rows with title, url, category, keywords, content, popularity, updated columns. No other data files needed.
WK 3 Baseline Search Engine — Student-Written From Scratch 3 hours · Most important week · Zero AI code generation ▼
Hr 10:00–1:00
🎙️ Faculty — Search Engine Concept
  • Explain how a search engine works using the librarian analogy: query comes in → scan every book card → score by relevance → return top 5
  • Explain the scoring formula on whiteboard: Title match = +10 points, Keywords match = +5 points, Content match = +2 points, Popularity ÷ 10 = bonus
  • Walk through a worked example: search "python programming" — show how 3 pages score differently and why the Python Wikipedia page wins
  • Introduce the 5 functions students will write: load_pages(), tokenise_query(), contains_ignore_case(), score_page(), print_results()
  • State the Golden Rule clearly: every line of search_engine.py must be typed by the student — no copy-paste from AI, no "write this function for me"
✏️ Student — Plan Before Coding
  • On paper: draw the flow of search_engine.py before writing any code — what does the program do step by step?
  • Write out the scoring formula in your notebook with the worked example from the whiteboard
  • Name the 5 functions you will write and what each one does
  • Open VS Code — create search_engine.py in your project folder
Hr 21:00–2:00
💻 Faculty — Live Code Walk (Explain Only)
  • Explain load_pages() concept — how csv.DictReader reads metadata.csv into a list of dictionaries
  • Explain tokenise_query() — how str.split() turns "python programming" into ['python', 'programming']
  • Explain contains_ignore_case() — why str.lower() is needed for case-insensitive matching
  • Explain score_page() — the loop structure, conditionals for title/keywords/content, arithmetic for popularity bonus
  • Explain print_results() — how sorted() with reverse=True orders by highest score first, how f-strings format the output
  • Do NOT write the code on projector — explain concepts only; students write their own version
💻 Student — Write Every Line
  • Write load_pages() — read metadata.csv, return a list of page dictionaries
  • Write tokenise_query() — split query string into a list of lowercase words
  • Write contains_ignore_case() — return True if a word appears in text, ignoring case
  • Write score_page() — calculate relevance score for one page against all query words
  • Write print_results() — display top 5 results sorted by score with URL, category, popularity
  • AI permitted for explanation only: "Explain what csv.DictReader does" — not "write this function for me"
Hr 32:00–3:00
🧑‍🏫 Faculty — Live Testing + Viva Prep
  • Ask each student to run their search engine live on the projector with one query
  • Check: does the output show 5 results? Do scores look correct? Are real URLs displayed?
  • Ask 2 spot-check questions per student: "What does score_page() do?" and "Why do you use str.lower()?"
  • Identify students who cannot explain their code — note for follow-up; they must re-write without AI assistance
⚠️ Golden Rule Reminder
  • If a student cannot explain any function line by line, their score for Week 3 is zero — re-submission required
  • Code was written by the student's hands — the viva will prove it
🧪 Student — Test + Explain
  • Run search_engine.py with at least 3 different queries — check results make sense
  • Test edge cases: what happens with a query that matches nothing? What if the query has one word?
  • Practise explaining score_page() out loud — you will be asked in the viva
  • Log any AI interactions used today in your Notion diary
📄 Week 3 Deliverable: search_engine.py — working baseline search engine that reads metadata.csv, accepts a query, scores 100 real pages, and displays top 5 results with real URLs. Student can explain every single line.
WK 4 AI-Augmented Ranking — 4 Smarter Ranking Methods 3 hours · GitHub Copilot introduced (refactoring only) ▼
Hr 10:00–1:00
🎙️ Faculty — Why Improve Ranking?
  • Explain why the baseline is limited: treats all pages the same regardless of source quality or recency — real search engines do not
  • Introduce the 5 ranking methods on projector: Weighted Field Ranker, Popularity-Aware Ranker, Keyword Count Ranker, Freshness-Aware Ranker, Domain Trust Ranker
  • Assign one method to each team (2 students per team)
  • Introduce GitHub Copilot — show what it does and what it does NOT do in this project
  • Demonstrate Copilot suggesting a better variable name and adding a docstring — show reviewing and accepting the suggestion
  • State clearly: Copilot does not write ranking logic — students write the logic, Copilot helps clean it up
📋 Student — Understand Your Method
  • Read the description of your assigned ranking method — understand what changes versus the baseline
  • Write in notebook: what is the one thing your method does differently, and what kind of query will it improve?
  • Copy search_engine.py → rename as search_engine_v2.py — this is where your improvement goes
  • Open GitHub Copilot in VS Code — try suggesting a docstring for one function from Week 3
Hr 21:00–2:00
🤖 Faculty — Ranking Method Guidance
  • Circulate — help each team implement their ranking method
  • Team 1 (Weighted Field Ranker): guide on multiplying title score by 1.5
  • Team 2 (Popularity-Aware): guide on changing popularity ÷ 10 to popularity ÷ 5
  • Team 3 (Keyword Count): guide on using count() instead of in operator for term frequency
  • Team 4 (Freshness-Aware): guide on parsing the updated column date and adding +5 bonus for pages updated within 90 days
  • Team 5 (Domain Trust): guide on checking if "wikipedia.org" or "geeksforgeeks.org" appears in the URL string
💻 Student — Implement + Refactor
  • Implement your ranking method in search_engine_v2.py — write the logic yourself
  • Use Copilot only for: renaming variables, adding docstrings, splitting long functions — not for writing the ranking logic
  • Every Copilot suggestion: accept or reject — document the decision in prompts-log.md with a reason
  • Run v2 with 3 queries — compare output to v1 (baseline) for the same queries
Hr 32:00–3:00
🗣️ Faculty — Team Sharing Round
  • Each team presents their method to the class in 5 minutes: what changed, what query it helps, one live demo
  • Class discussion: which method seems most promising and why?
  • Remind students: next week they will test all 5 methods with 6 standard queries and record evidence
🗣️ Student — Present Your Method
  • Present your team's method to the class — 5 minutes maximum
  • Run one live query on the projector showing the v2 result vs v1 result
  • Update prompts-log.md with all Copilot interactions from today
📄 Week 4 Deliverable: search_engine_v2.py — 5 ranking methods (baseline + 4 improvements). Student can explain what changed in each method and why. prompts-log.md updated with all Copilot interactions.
WK 5 Testing & Comparison — Evidence-Based Evaluation 3 hours · 6 standard queries · comparison-table.xlsx ▼
Hr 10:00–1:00
🎙️ Faculty — Why Test?
  • Explain: in industry, no algorithm is deployed without evidence — Week 5 teaches students that better code requires data, not guesswork
  • Introduce the 6 standard test queries (same for all students): python programming language · machine learning artificial intelligence · punjab tourism chandigarh · data structures algorithms · career internship jobs india · cricket india sports
  • Share the comparison-table.xlsx template — show the 4 columns to fill: Relevance (1–5), Speed (ms), Readability (1–5), Scalability
  • Demonstrate how to measure speed using time.perf_counter() — run it live on projector
  • Explain relevance rating: do the top 5 results make sense for the query? Rate honestly 1–5
📋 Student — Set Up Test Sheet
  • Open comparison-table.xlsx template from LMS
  • Add time.perf_counter() to your search_engine_v2.py to measure speed per query
  • Write down the 6 standard queries in your notebook — you will run each query on all 5 methods
  • Plan: 5 methods × 6 queries = 30 tests to run this session
Hr 21:00–2:00
🧑‍🏫 Faculty — Guided Testing Round
  • Circulate — ensure students are running all 6 queries on all 5 methods (not skipping)
  • Spot-check: ask one student to justify their relevance rating — does the reasoning make sense?
  • Flag if any student has all relevance scores as 5 — require them to re-test and rate more critically
  • Introduce peer readability review: swap code with a classmate, rate the ranking code readability 1–5
🧪 Student — Run All 30 Tests
  • Run all 6 queries on all 5 ranking methods — record results in comparison-table.xlsx
  • For each test: note the top 5 results, rate relevance 1–5, record speed in ms
  • Swap code with a classmate — rate their ranking method's readability 1–5 and explain your rating
  • Assess scalability conceptually: would this method still work with 10,000 pages? Write one sentence per method
Hr 32:00–3:00
📊 Faculty — Analysis Discussion
  • Show class results on projector: which method scored highest for relevance across the 6 queries?
  • Discuss as a class: does speed matter for 100 pages? What about 10,000 pages?
  • Introduce inverted index concept: explain why real search engines do not loop through every page per query
  • Ask each student: "Which method would you deploy in a real product and why?" — students write their answer
✍️ Student — Form Your Opinion
  • Complete comparison-table.xlsx with all 30 test results and peer readability scores
  • Write a 100-word conclusion in the notes section: which method works best, which is most readable, which would scale best and why
  • Save final comparison-table.xlsx — this is a graded deliverable
  • Update Notion AI diary if any AI tools were used today
📄 Week 5 Deliverable: comparison-table.xlsx — side-by-side results for all 5 methods across 6 standard queries. Relevance, speed, readability, and scalability filled with real data — no guesses. Written conclusion included.
WK 6 Report, Viva & Live Demonstration 3 hours · Final submission · 15-mark viva ▼
Full Session0:00–3:00
🎤 Faculty — Viva Conduct
  • Call students one by one — each viva is approximately 15 minutes: 5-minute live demo + 10 minutes of questions
  • During demo: student runs search engine in terminal, types 2 queries live, shows results with real URLs
  • Ask 5–8 questions per student — drawn from the viva question bank below
  • Score on rubric: Web Crawler (2 marks) · Baseline Engine (3 marks) · Improved Ranking (3 marks) · Comparison Table (3 marks) · AI Log (2 marks) · Viva & Demo (2 marks) = 15 marks total
  • Use remaining time for group debrief — share top 3 technical insights from class
🔍 Viva Question Bank
  • What does your crawler do? Explain requests.get(), BeautifulSoup, time.sleep().
  • Why do you not save the HTML files to disk?
  • How does your score_page() function work? Walk me through the scoring formula.
  • What was the difference between your baseline and your improved method? Give specific numbers from your tests.
  • What would happen with 10,000 pages instead of 100? What is an inverted index?
  • Which AI tools did you use and what exactly for? Show me a specific entry from your prompts-log.md.
  • What does BeautifulSoup do? Explain find() and get_text().
  • Why do we need metadata.csv? What would happen without it?
🖥️ Student — Present + Defend
  • Submit all project files via Google Form before the session begins — link shared by faculty on WhatsApp
  • Submit: url-list.txt, crawler.py, metadata.csv, search_engine.py, search_engine_v2.py, prompts-log.md, comparison-table.xlsx, report.md — zipped in one folder
  • Bring laptop with search engine ready to run in terminal
  • Demo flow: open terminal → run search_engine_v2.py → type "python programming" → show top 5 results → type a second query of faculty's choice
  • Answer viva questions — be specific, not generic: say "I used ChatGPT in Week 2 to understand why my GeeksForGeeks URL returned 403" not just "I used ChatGPT to help"
  • After your viva: watch peers present and write 1 peer observation note
📋 Final Submission Checklist
  • ✅ url-list.txt — 100 URLs, one per line
  • ✅ crawler.py — fetches and extracts from real websites
  • ✅ metadata.csv — 100 rows, all columns filled
  • ✅ search_engine.py — baseline, student-written
  • ✅ search_engine_v2.py — 5 ranking methods
  • ✅ prompts-log.md — minimum 15 logged AI interactions
  • ✅ comparison-table.xlsx — 30 real test results
  • ✅ report.md — 8–10 pages
📄 Week 6 Final Submission: Complete project ZIP · report.md (8–10 pages) · live demo in terminal · viva (15 min per student) · 15-mark rubric score.
